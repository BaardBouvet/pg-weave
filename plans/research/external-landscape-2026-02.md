# External Research: Malloy, BigQuery Pipe SQL, OSI/MetricFlow

> **Date:** 2026-02-26  
> **Status:** Internal analysis

## Scope

This note reviews three ecosystems and extracts ideas that may improve `pg_weave`:

- Malloy (`malloydata.dev` + docs)
- BigQuery Standard SQL Pipe Syntax
- Open Semantic Interchange (OSI) / MetricFlow semantic model

The goal is not feature parity. The goal is to identify high-value, low-friction ideas that fit `pg_weave`'s current SQL-first and dialect-neutral direction.

## Snapshot of each project

### Malloy

What stands out:

- Clear split between **semantic model definitions** and **query blocks**.
- Strong emphasis on **named outputs** (`name is expr`) for readability.
- Reusable source-level declarations for joins/measures/dimensions.
- Built-in support for nested result structures and multi-stage pipelines.
- Explicit handling of aggregate correctness in join-heavy scenarios.

Transferable signal for `pg_weave`:

- Keep prioritizing composable modeling + concise query syntax.
- Prefer syntax that makes output shape obvious.
- Keep nested-output ergonomics as a differentiator.

### BigQuery Pipe Syntax

What stands out:

- Uniform `|>` pipeline where each operator is a self-contained table transform.
- Strong **operator decomposition** (`EXTEND`, `SET`, `DROP`, `RENAME`, `AGGREGATE`, etc).
- Clear locality rules: each stage works from immediate input.
- Mix-and-match compatibility with standard SQL.

Transferable signal for `pg_weave`:

- Operator-level transformations improve readability and refactorability.
- Stage semantics reduce hidden coupling and make validation easier.
- SQL interop should remain a first-class requirement.

### OSI / MetricFlow

What stands out:

- Vendor-neutral semantic contract (`entities`, `dimensions`, `measures`, `metrics`).
- Explicit semantic graph with traversal via entity join keys.
- Typed metric definitions (simple, ratio, derived, cumulative, conversion).
- Portability and governance as product goals, not afterthoughts.

Transferable signal for `pg_weave`:

- A minimal semantic contract can increase portability and AI/BI interoperability.
- Explicit entity semantics can improve graph traversal correctness.
- Metric abstraction may be valuable, but only if kept optional and incremental.

## Ideas worth adopting (near-term)

1. **Keep and sharpen stage-local semantics**
   - Aligns with BigQuery pipe behavior and current `pg_weave` direction.
   - Validation becomes more precise because each block has explicit input/output.

2. **Strengthen output-shape explicitness**
   - Malloy’s “everything named” principle is useful for readability and downstream stability.
   - For `pg_weave`, this suggests stronger guidance and linting before adding syntax.

3. **Introduce optional semantic annotations (lightweight)**
   - Borrow OSI’s `entity/dimension/measure` framing without adopting YAML-first workflows.
   - Keep SQL/DSL as source of truth; annotations enrich tooling, docs, and validation.

4. **Codify transform primitives as stable vocabulary**
   - BigQuery-style decomposition maps well to existing `SET`, `WHERE`, `COLLECT` patterns.
   - Prefer explicit primitives over overloaded “do everything” constructs.

## Ideas to avoid (for now)

1. **Full external semantic spec compatibility as immediate goal**
   - High implementation scope and unclear user value at current project maturity.

2. **Hard requirement for metric-layer abstraction before core DSL stabilizes**
   - Could slow core language convergence.

3. **Large syntax expansion without operator-level invariants**
   - Increases parser/compiler complexity without guaranteed usability gains.

## Candidate ADR topics generated by this research

1. **Semantic Annotations (Optional, Inline)**
   - Define whether `pg_weave` should support optional entity/dimension/measure metadata.
   - Clarify compiler responsibilities vs documentation/tooling responsibilities.

2. **Stage Locality and Visibility Rules**
   - Formalize what each block/stage can reference (immediate input vs outer scopes).
   - Make this explicit for predictable validation and SQL generation.

3. **Output Contract Strictness**
   - Decide if the language should enforce stronger naming/output-contract guarantees.
   - Evaluate lint-only vs compile-time enforcement.

## Fit against existing direction

- Reinforces accepted decisions around dialect neutrality and SQL pass-through.
- Supports graph traversal work by emphasizing explicit entity relationships.
- Suggests adding semantic metadata incrementally rather than introducing a second full modeling language.

## Proposed next steps

1. Draft a short ADR: **Stage Locality and Visibility Rules** (high confidence, low ambiguity).
2. Add a design sketch (non-ADR note) for **optional semantic annotations** with 2-3 syntax options.
3. Validate against current examples in `docs/` and `plans/dtl/` to ensure no regression in readability.

## Bottom line

The strongest external ideas for `pg_weave` are:

- BigQuery’s stage-oriented operator semantics,
- Malloy’s output-shape clarity and model/query separation,
- OSI/MetricFlow’s explicit semantic contracts.

The pragmatic path is to adopt these as **incremental constraints and metadata**, not as a wholesale language or workflow pivot.
